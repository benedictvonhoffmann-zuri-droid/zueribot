## TL;DR — Top 5 actionable findings

1. **Entity confusion: Hive (Club) → Freibad Heuried.** Jonas asked "Isch d Hive no offe?" and the bot answered about Freibad Heuried being a seasonal pool. Register=5 but the answer is about a completely different venue. This is the single most damaging failure in the run.
2. **Language-register drift to Hochdeutsch** is the most frequent pattern (9/16 = **register avg 3.25/5**). Even when the user writes Züridütsch, bullet-point formatted answers slip into Hochdeutsch (sofia_student weather reply). Priya_tourist (English-in/English-out) was also flagged register=2 — judge noise there, but the overall signal is real.
3. **Unsourced real-time claims.** Ana got "Letten isch grad 12°C" with a citation but no connector trace proves a call was made. Markus asking about Sihlcity got "60% belegt" for *other* parking garages presented as if current. Prompt lacks a hard rule forbidding specific numbers without a connector call in the same turn.
4. **Empty-KB fallback not triggering (3×).** Eventfrog gave no price data → bot told user to go to eventfrog.ch instead of calling `web_search`. Same pattern for Hive/nightlife. The prompt *already* contains the rule ("never say I couldn't find anything without web_search") — so this is either an obedience issue or the agent isn't re-entering the tool loop after a negative result. Worth investigating in the agent, not just the prompt.
5. **Connector coverage gaps.** `get_parking` does not include Sihlcity (one of Zürich's busiest garages); `get_events` returns no price field; nightlife/club queries have no connector at all. These are all fixable by either expanding the connector contracts or adding explicit fallback rules.

## System-Prompt-Diffs

Target file: `backend/agent.py`, the `SYSTEM_PROMPT = """…"""` block.

### Diff 1 — Add a hard rule forbidding unsourced real-time values

Reason: three separate scenarios (Letten temp, Sihlcity parking, weather) cited connectors but the judge flagged the realism. A clearer rule reduces the temptation to interpolate.

```diff
 - When mentioning times, use 24h format
+- **Realtime-values rule**: never state a specific current number (temperature,
+  occupancy %, free spaces, wait minutes, price) unless a connector call
+  *in this same turn* returned that exact value. If no connector was called
+  or it failed, write "Ich habe für X gerade keine Live-Daten" and suggest
+  the source the user can check themselves. Never extrapolate from a related
+  value (e.g. other parking garages, yesterday's temperature).
 - For questions about specific places (restaurants, shops, venues): call `search_knowledge_base` for editorial context AND `get_pois` or `get_venues` for current addresses/hours — both in the same turn
```

### Diff 2 — Disambiguate name collisions

Reason: Hive (club) ↔ Heuried (Freibad) collapsed into the same answer. A general rule beats a special case.

```diff
-- **When the user asks about a specific named restaurant or venue** (e.g. "Vallocaia", "Kronenhalle"): call ALL THREE in parallel — `search_knowledge_base` with the name, `get_pois` with query=restaurant_name (name search, not category), AND `get_venues` with name_filter=restaurant_name and category=gastronomy. If all three return no results, immediately follow up with `web_search("{name} Zürich Adresse Öffnungszeiten")`.
+- **When the user asks about a specific named restaurant, club, bar or venue** (e.g. "Vallocaia", "Kronenhalle", "Hive", "Rote Fabrik"): call ALL THREE in parallel — `search_knowledge_base` with the name, `get_pois` with query=venue_name (name search, not category), AND `get_venues` with name_filter=venue_name. If all three return no results, immediately follow up with `web_search("{name} Zürich")`.
+- **Name-match discipline**: if the retrieval returns a venue whose name is only *similar* to the one the user asked for (e.g. user says "Hive", result is "Heuried"; user says "Rote Fabrik", result is "Roter Turm"), treat that as a miss, NOT a match. Do not answer about the similar-but-wrong venue. Fall through to `web_search` and tell the user you couldn't find their exact venue.
```

### Diff 3 — Strengthen the language-lock rule

Reason: the existing rule ("respond in same language") is buried at the top of "Rules" and gets overridden as soon as the answer becomes structured. Make it explicit and give concrete markers for Züridütsch.

```diff
-- Always respond in the same language the user writes in (German, Swiss German, English, French, Italian)
-- If the user writes in Swiss German, respond in Swiss German
+- **Language-lock rule**: detect the user's language from their FIRST message
+  and lock it for the whole conversation, including bullet points, tables,
+  and structured answers.
+  - Züridütsch in → Züridütsch out. Use: *chunnt, isch, git, hüt, wänn, händ,
+    öppis, nöd, au, sust, bi mir*. Avoid: *kommt, ist, gibt, heute, wenn,
+    haben, etwas, nicht*.
+  - Hochdeutsch in → Hochdeutsch out, no Swiss words.
+  - English in → English out, no German phrases mixed in.
+  If you catch yourself drifting mid-reply (e.g. a dialect question got a
+  Hochdeutsch bullet list), rewrite the reply before sending.
```

### Diff 4 — Tighten the empty-KB fallback

Reason: the rule already exists at line 67 of the prompt but is being ignored in practice. Make it unignorable.

```diff
-| KB returns empty or clearly irrelevant results | Follow up immediately with `web_search` — never say "I couldn't find anything" without first trying web_search |
+| KB returns empty or clearly irrelevant results | You MUST call `web_search` in the same turn. Do not: (a) tell the user to check an external website themselves, (b) list possible websites, (c) ask the user to clarify — those are forbidden escapes. Call web_search first, then decide. |
```

## Connector changes

### 1. `get_parking` — Sihlcity missing
- **Gap:** user asks about one of Zürich's three busiest garages; connector returns unrelated others.
- **Options (pick one):**
  - (a) Expand connector to include Sihlcity's own feed (APCOA-operated; may require separate scraper).
  - (b) If (a) isn't feasible, add to system prompt: _"If the user asks about a specific named parking garage (Sihlcity, Jelmoli, Globus, Opéra, Bellevue), and it's not in the get_parking result, call web_search immediately — do not list unrelated garages as 'alternatives'."_
- **Priority:** higher than `get_pois` for parking queries.

### 2. `get_events` — no price field exposed
- **Gap:** user asks "events unter 20 Franken"; connector can't filter → bot deflects.
- **Fix:** check if the Eventfrog API returns a price field `??`. If yes, add `min_price`/`max_price` to the tool schema. If no, update the tool description so the agent knows not to promise price filtering and instead lists events + tells the user prices must be checked on the event page.

### 3. Nightlife / club coverage
- **Gap:** no connector fires for "is Hive open tonight". Judge flagged 2×.
- **Options:**
  - (a) Extend `get_events` with `venue_name_filter`.
  - (b) Add `get_venues` call for category=nightlife with opening hours if the zuerich.com API exposes that `??`.
  - (c) Prompt rule: for any nightlife/club query, always call `web_search("{venue} Zürich tonight")` alongside the existing tools.

### 4. `get_water_temps` — Limmat / Letten coverage
- **Gap:** bot gave "12°C" without a visible connector call. Either the connector doesn't include the Letten (unlikely to return data at all in April) or it was called but the bot added a value after a failed call.
- **Fix:** verify `get_water_temps` covers Unterer/Oberer Letten. If it doesn't, add Limmat Wasserwerk measurement point. Either way, tighten the connector-failure handling in `agent.py` so an empty result does not get turned into a specific number.

## Things to investigate — not prompt fixes

These showed up in the data but are likely **agent-loop / retrieval bugs**, not prompt wording issues. Don't paper over them with prompt edits:

- **Empty-KB fallback non-triggering** — the rule exists but the agent doesn't re-enter the tool loop. Check `backend/agent.py` `should_continue` / `grade_rag_results` logic. Possibly the grader returns `relevant=true` for "no results" messages because they're long and structured.
- **Hive → Heuried collapse** — this smells like embedding-retrieval collapsing two similar-sounding venue names. Worth checking the embeddings index and whether `get_pois`/`get_venues` are actually being called with `"Hive"` or getting fuzzy-matched to "Heuried".
- **Judge noise on Priya's English replies** (register=2 while speaking correct English) — not a Bünzli bug, a judge-prompt bug. In v2, split "language match" into a binary check computed programmatically (see [CONCEPT_V2.md](../CONCEPT_V2.md) §5).
